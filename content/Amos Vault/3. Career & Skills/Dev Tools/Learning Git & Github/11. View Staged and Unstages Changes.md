## Overview
- Before we commit what we have in our staging area, we need to <span style="color:rgb(255, 148, 148)">review our code.</span>
- We do not want to commit<span style="color:rgb(255, 148, 148)"> bad code </span>or broken code into our repository.
- The <span style="color:rgb(255, 148, 148)">best practice </span>is to review what we have in the staging area before commiting.


#### To see changed in the Staging Area.
So, git status allows us to see the files that we have changed. But how can we see the <span style="color:rgb(255, 255, 0)">exact lines of code</span> that we have staged?
<span style="color:rgb(0, 176, 80)">Answer</span> : We use the <span style="color:rgb(255, 255, 0)">diff </span>command.
ie:
```bash
git diff --staged
```

Output:
![[Pasted image 20250901104916.png | 300]]
From the results: The call from <span style="color:rgb(255, 0, 255)">the diff utility </span>has been made.
> two files in the staging area are being compared.
> a/filex.txt->original file
> b/filex.txt->copy of the riginal file
> -1 : Means starting from the first line of a/filex.txt, one line has been extracted and shown.
> +1,5 : Means starting from the first line of b/filex.txt five lines have been extracted and displayed
> - Lines with the +  signs, show the lines that we have added in the new copy. We have added these lines in the staging area.


#### To see the changes in the working directory that have not yet been stages yet.
- Here we use the diff command, without any arguments.
ie:
```bash
git diff
```

Ouput:
![[Pasted image 20250901110128.png | 300]]
From the output:
> We are comparing two file copies
> a/filex.txt -> The file that we currently have in our working directory.
> b/filex.txt-> The file that we alredy had in the staging area.
> -1,5  : Means from the first line of oldfile, five lines have been extracted and displayed.
> +1,5 : Means from the first line of new copy, five lines have been extracted and displayed.
> +words : Means the words that have been added to a file.
> -words : Means the words that have been deleted from the file.


#### Summary
- <span style="color:rgb(255, 255, 0)">git diff</span> : Displays the unstaged changes.
- <span style="color:rgb(255, 255, 0)">git diff --staged</span> : Displays the staged changes that will go into the next commit.


## Using Visual Tool for compari2ng files.
- Instead of using the terminal to observe stages and unstaged changes in the files, we are going to use visual tools for simplicity.

Options Available are :
- KDff3
- P4Merge
- Win Merge(Windows Only)
- VSCode (We Shall use this)

##### Steps for using VSCode as the default diff tool.
1. Configure diff global setting
```bash
git config --global diff.tool vscode
```
NB : The global settings applies to all our repositories. 

2. We need to tell git how to launch vs code
```bash
git config --global difftool.vscode.cmd "code --wait --diff $LOCAL $REMOTE"


//Which worked for me
git config --global difftool.vscode.cmd "code --wait --diff \"$LOCAL\" \"$REMOTE\""

```

OR: Open <span style="color:rgb(255, 255, 0)">.gitconfig </span>in vscode  and set as follows
![[Pasted image 20250901114935.png | 300]]
From the above :
> i. code --wait : Tells the window to wait until we are done with using vs code.
> So the terminal will wait until we have closed the vscode instance.
> ii. --diff : Tells vscode that we want to us it for diffing or comparing files.
> iii. $LOCAL $REMOTE : These are the place holders for the Old and new copies of our file.

3. Prove to see if configurations were set.
- By opening the configurations file in the default edito
ie:
```bash
git config --global -e
```
- You can edit anything if you want.

4. To view unstages changes
```bash
git difftool
```
- So, We are using difftool command instead of just diff

5. To view  the staged changes
```bash
git difftool --staged
```

- When there is no output, mean all the changes in our working directory have been staged.

<span style="color:rgb(255, 148, 148)">Staged Comparison</span> : What we have in the <span style="color:rgb(255, 255, 0)">Staging Area </span>vs What we have in the <span style="color:rgb(255, 255, 0)">Last Commit</span> .
